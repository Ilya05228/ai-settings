# Java best practices

## Когда действует

Как только в запросе фигурируют Java-исходники, фрагменты на Java, `.java`, или ты собираешься их выдать — считай этот skill **выбранным по умолчанию** и следуй ему целиком, даже если в интерфейсе не отмечен отдельный skill.

Считай правила ниже **обязательными**, как только в задаче есть генерация, правка или обсуждение Java-исходников.

## Явные конструкторы вместо Lombok (временное правило)

**Сейчас не используй** аннотации Lombok (и аналоги), которые генерируют конструкторы, `equals`/`hashCode`, геттеры/сеттеры или «замораживают» поля на основе списка полей, в том числе:

- `@Value`, `@Data`, `@AllArgsConstructor`, `@RequiredArgsConstructor`, `@NoArgsConstructor`, `@Builder` (и сочетания), если они заменяют то, что ты мог бы написать явно

**Вместо этого:** пиши **ручные конструкторы** (и при необходимости фабрики, статические factory-методы), явно инициализируй поля, сам контролируй инварианты. То же для `equals`/`hashCode`/`toString`, если они нужны — явная реализация или честный выбор API без магии генерации.

Это ограничение ослабят позже; пока оно действует как жёсткое.

## Остальная база (кратко)

- Предпочитай читаемость и предсказуемость: понятные имена, короткие методы, ранний выход, без лишней вложенности.
- Не подавляй предупреждения и исключения без причины; не глотай `InterruptedException`.
- Для публичных API документируй контракт там, где это принято в проекте (`javadoc` или нет — как у остального кода).
- Следуй стилю существующего модуля (форматирование, Optional, checked exceptions — как уже принято).

Когда пользователь явно просит DDD, Clean Architecture или доменные события, **дополнительно** согласуйся с skills `ddd`, `clean-architecture`, `domain-events` из этого же плагина — они про архитектуру, этот skill про язык и базовые Java-соглашения.
