# Domain Events

Используй этот skill, когда задача отдельно касается доменных событий, их смысла, формы, буферизации и публикации.

## Что такое доменное событие

- `DomainEvent` отражает существенный факт предметной области, а не технический шаг
- событие сообщает, что в домене уже произошло значимое изменение
- событие не должно быть просто копией всего агрегата без необходимости

## Когда создавать события

- не создавай события при создании агрегата по умолчанию, если это не требуется бизнес-правилом
- событие создаётся только для действительно значимых изменений состояния
- событие должно отражать изменённые свойства или смысл изменения, а не тащить весь агрегат целиком

## Где живут события

- источник события - агрегат или доменный сервис
- не создавай доменные события в контроллерах, transport-слое, ORM-слое или случайных утилитах
- если в проекте уже есть базовый контракт `DomainEvent`, следуй ему

## Буфер событий агрегата

- агрегат хранит внутренний буфер событий
- `record(event)` регистрирует событие внутри агрегата или доменного сервиса по правилам проекта
- `pullDomainEvents()` возвращает накопленные события и очищает буфер
- события публикуются после сохранения агрегата, а не до него

## Публикация

- `EventBus` публикует список событий после сохранения агрегата
- типовая форма публикации: `EventBus.publish(events)`
- порядок простой: изменить агрегат -> сохранить агрегат -> забрать `pullDomainEvents()` -> опубликовать события

## Структура события

- событие должно содержать только то, что нужно для передачи доменного факта
- если у проекта есть базовые поля вроде `aggregateId`, `occurredOn` и аналоги, используй их
- не раздувай событие инфраструктурными деталями и лишним payload

## Чего не делать

- не создавай событие на каждое тривиальное изменение без предметного смысла
- не публикуй событие до сохранения агрегата
- не превращай доменное событие в транспортный DTO
- не тащи в событие весь агрегат, если можно выразить изменение меньшим набором данных

## Проверка перед завершением

- событие выражает предметный факт
- событие создаётся только при значимом изменении
- буфер событий хранится в агрегате
- `pullDomainEvents()` очищает буфер после выдачи
- публикация идёт после сохранения через `EventBus`
