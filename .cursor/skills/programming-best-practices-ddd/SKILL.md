# DDD

Используй этот skill только когда пользователь явно хочет применять DDD-подход или когда этот skill уже подключен в текущей задаче.

## Базовый принцип

- если skill подключен, строй решение вокруг предметной области, а не вокруг фреймворка, таблиц, DTO или transport-слоя
- модель должна отражать язык бизнеса и значимые понятия домена
- инфраструктура подстраивается под домен, а не наоборот

## Основные типы

Используй и различай следующие роли. Если проект уже содержит общий доменный каркас, новые типы обязаны встраиваться именно в него.

- `ValueObject` / `Value Object` - базовый тип для объектов-значений без собственной идентичности
- `IdentifierInterface<V>` - базовый контракт идентификатора
- `LongIdentifier` - базовый id поверх `Long`, если проект использует long-based id
- `UuidIdentifier` / `UUIDIdentifier` - базовый id поверх `UUID`, если проект использует uuid-based id
- конкретные id вроде `UserId`, `OrderId`, `ProjectId` - наследники базового идентификатора, а не сырые типы
- `Entity<ID extends IdentifierInterface<?>>` - сущность с устойчивой идентичностью и жизненным циклом
- `Aggregate<ID extends IdentifierInterface<?>>` - наследник `Entity`, корень агрегата и граница консистентности
- `Repository<A, ID>` - контракт загрузки и сохранения агрегата, а не произвольной внутренней сущности
- `DomainService` - доменная операция без собственного состояния, если логика не принадлежит одной сущности
- `UseCase` / `Interactor` - прикладной сценарий, который оркестрирует домен, но не заменяет его

Не смешивай эти роли между собой и не подменяй их случайными универсальными классами.

## Базовая иерархия типов

Ориентируйся на такую форму каркаса, если в проекте нет уже существующего совместимого варианта:

- `ValueObject`
- `IdentifierInterface<V>`
- `LongIdentifier` и/или `UuidIdentifier`
- конкретные id как наследники базового идентификатора
- `Entity<ID extends IdentifierInterface<?>>`
- `Aggregate<ID extends IdentifierInterface<?>>` как наследник `Entity`
- `Repository<A, ID>`
- `DomainService`

Если в проекте уже есть похожая иерархия, используй её названия и базовые типы, а не создавай параллельную.

## Value Objects вместо примитивов

- для значимых бизнес-понятий создавай `Value Object`, а не используй сырые примитивы и строки
- не передавай в доменную модель голые `String`, `Long`, `UUID`, `Integer`, `BigDecimal` и похожие типы, если у значения есть предметный смысл
- вместо сырого значения создавай тип с именем доменного понятия

Примеры:
- не `String email`, а `Email`
- не `Long userId`, а `UserId`
- не `BigDecimal amount`, а `Money` или другой осмысленный тип
- не `Integer quantity`, а `Quantity`

Если встроенный тип всё же появляется на границе системы, быстро оборачивай его в доменный тип перед работой в доменной модели.

## Идентификаторы

- каждый id оформляй отдельным типом, а не используй голые идентификаторы
- если в проекте уже есть базовый контракт вроде `IdentifierInterface`, `LongIdentifier`, `UuidIdentifier` или аналог, используй его
- если общий каркас уже существует, новые `OrderId`, `UserId`, `ProjectId` и другие id должны наследоваться от него или реализовывать его
- если каркаса нет, создай строгий и понятный базовый тип идентификатора и строй id поверх него
- по умолчанию предпочитай отдельный базовый id и его специализации, а не прямое наследование всех id от произвольного utility-типа

По умолчанию считай, что сырой id не должен протекать по доменной модели.

## Наследование от базовых доменных типов

- если в проекте уже есть `ValueObject`, `IdentifierInterface`, `LongIdentifier`, `UuidIdentifier`, `Entity`, `Aggregate`, `Repository`, `DomainService` или похожие базовые абстракции, новые типы обязаны встраиваться в этот каркас
- не создавай параллельные несовместимые иерархии вроде "ещё один EntityBase", если в проекте уже есть принятый тип сущности
- не создавай отдельный базовый id, если в проекте уже есть принятый `LongIdentifier` или `UuidIdentifier`
- `Aggregate` должен использовать тот же базовый id-каркас, что и `Entity`
- сначала ищи shared kernel, common-модуль, domain-common, core-types или аналогичный слой общих доменных типов
- если общий каркас не найден, строй новый строго и последовательно: базовые доменные типы должны быть явными, а не размазанными по случайным файлам

## Сущности и агрегаты

- каждая `Entity` должна принадлежать одному агрегату или сама быть его корнем
- `Aggregate` - это специальный вид `Entity`, а не независимая параллельная сущность
- репозиторий должен работать с корнями агрегатов, а не с произвольными внутренними сущностями
- не обходи границы агрегата ради удобства загрузки или обновления части модели
- если инвариант относится к нескольким внутренним объектам, его должен контролировать агрегат

## Неизменяемость

- делай доменные модели неизменяемыми там, где это уместно по языку и стилю проекта
- если состояние надо "изменить", лучше вернуть новый экземпляр или применить явно выраженную доменную операцию
- не раздавай сеттеры только ради удобства ORM или сериализации

## Final по умолчанию

- конкретные реализации делай `final` по умолчанию, если расширение не задумано специально
- наследование считай осознанной привилегией, а не стандартом
- абстрактные базовые типы, интерфейсы и осознанные extension points не нужно делать `final`

## Null и optional

- не используй `null` как основной способ выражения отсутствия значения
- если язык и стиль проекта поддерживают `Optional`, используй его для действительно опциональных полей
- не размазывай null-проверки по домену, если проект уже решает этот вопрос типами, анализаторами или контрактами

## Доменные ошибки

- выражай ошибки домена отдельными типами, а не строками и универсальными исключениями
- если в проекте уже есть базовый тип вроде `BusinessError`, используй его как основу для доменных ошибок
- имя ошибки должно отражать смысл проблемы предметной области
- данные, важные для объяснения причины, храни в самой ошибке, а не теряй в свободном тексте
- текст сообщений и пользовательские формулировки держи на русском, если проект не требует иного

## Репозитории

- `Repository<A, ID>` описывает работу с агрегатами
- не смешивай в репозитории бизнес-правила, SQL-детали, HTTP и orchestration-логику
- сначала допускается in-memory реализация, затем production-реализация поверх реального хранилища

## Дженерики и типизация

- предпочитай явные generic-типы
- не злоупотребляй wildcard, если можно выразить тип строго и понятно
- сигнатуры должны помогать модели быть безопасной и читаемой, а не прятать смысл
- для `Entity`, `Aggregate`, `Repository`, `IdentifierInterface` и связанных базовых типов используй явные generic-параметры

## Описание классов

- если язык и стиль проекта это поддерживают, добавляй короткое описание класса или интерфейса над объявлением
- в описании указывай, что означает тип и за что он отвечает

## Чего не делать

- не смешивай доменную модель с ORM-аннотациями, transport DTO и framework-деталями, если это можно разделить
- не подменяй доменные типы "универсальными" map/object/blob структурами
- не используй примитивы там, где у значения есть отдельный бизнес-смысл
- не ломай существующий доменный каркас проекта, если он уже есть

## Связь с Clean Architecture

- этот skill отвечает за модель предметной области и её типы
- этот skill не объясняет подробно доменные события: для этого используй отдельный skill `domain-events`
- если задача ещё требует строгого разделения слоёв, портов, адаптеров и направления зависимостей, подключай также skill `clean-architecture`

## Проверка перед завершением

- у модели есть явный базовый `ValueObject`
- у модели есть базовый контракт идентификатора и специализации вроде `LongIdentifier` или `UuidIdentifier`
- доменные понятия выражены отдельными типами, а не примитивами
- id типизированы
- `Entity` использует базовый id-контракт
- `Aggregate` наследуется от `Entity` и использует тот же id-контракт
- `Repository` работает с агрегатами через явные generic-типы
- новые типы встроены в существующий domain-common каркас, если он есть
- агрегаты и репозитории не нарушают границы модели
- домен не зависит от инфраструктурных деталей без крайней необходимости
